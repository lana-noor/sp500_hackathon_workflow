Input Data Files (sp500_sample_data/)

tickerlist.csv

505 rows · Columns: index, name, ticker, sector

Maps every S&P 500 constituent to one of the 11 GICS sectors. Used by Agent 2 to roll up monthly returns from individual tickers into equal-weighted sector time-series.

returns.csv

60 rows (months) · ~481 columns
Columns: Date + one column per ticker

Monthly returns in decimal form (e.g. 0.0477 = 4.77%). Date range: Jan 2017 – Dec 2021. Primary source for all return-based metrics.

prices.csv

~62 rows · ~480 columns
Columns: one per ticker, no Date column

Monthly price snapshots in USD. Used for price-level analysis. Row index is implicit (no Date column — row order matches time sequence).

Agent Details
1
Query Planner
FoundryChatClient · No tools · Inline (no portal deployment needed)
FoundryChatClient No tools Pure reasoning

Takes the raw user question and converts it into a fully structured research plan that every downstream agent will use as their authoritative brief. It does not fetch data, run code, or search the web — it only reasons about what needs to be computed and why.

It identifies the analytical intent (sector comparison, risk screen, correlation study, etc.), determines the relevant sectors or tickers, scopes the time window within the available Jan 2017–Dec 2021 range, writes exactly 3–5 quantitative questions answerable from the CSVs, writes 2–4 qualitative questions requiring external web sources, specifies the chart requests with unique IDs and types, and sets the presentation angle (the story the final deck should tell).

"How did Information Technology compare
 to Energy from 2017 to 2021, and what
 drove the differences in performance?"

2
Quantitative Analyst
FoundryAgent · Code Interpreter tool · Deployed in Azure AI Foundry portal with CSV files attached
FoundryAgent Code Interpreter tickerlist.csv returns.csv prices.csv

The most computationally intensive agent. Receives the research plan and executes every quantitative question by writing and running Python code against the three CSV files in its secure sandbox. It produces structured findings (real computed numbers for each question) and chart_specs — data dictionaries containing the actual x/y series values that the local render_chart_specs() function will turn into PNG charts.

Standard analytical patterns it follows: sector roll-up (equal-weight mean of constituent monthly returns), cumulative returns via (1+r).cumprod()-1, max drawdown (peak-to-trough on cumulative curve), Sharpe ratio (annualised, 0% risk-free). It does NOT produce PNG files — only data dicts — so charts render reliably outside the sandbox.

Output is validated by _validate_agent2_output() before being saved to the checkpoint. If placeholder text, a missing chart_specs key, or empty series data is detected, the run fails immediately rather than saving bad data.

Output Validation — _validate_agent2_output()

Run automatically before saving to checkpoint. Raises ValueError and blocks the checkpoint save if any check fails.

  • Output must be valid JSON
  • Placeholder strings detected: "calculated value", "[number_of_rows]", "placeholder", etc.
  • Missing required keys: findings, chart_specs
  • findings dict is empty or all values are null
  • chart_specs is empty list (no charts produced)
  • chart_spec missing series key, or series has no data points
3
Search Query Expander
FoundryChatClient · No tools · Inline (no portal deployment needed)
FoundryChatClient No tools Pure reasoning

Bridges the quantitative world (Agent 2) and the web research world (Agent 4). It receives the qualitative questions from the research plan and the data summary from Agent 2 — the actual numbers — and converts them into 4–8 precise, targeted web search query strings.

The key insight: because it knows what Agent 2 actually found (e.g. Energy returned -58% max drawdown), it can write grounded, specific queries rather than generic ones. It targets different source types per query: news for events, analyst reports for commentary, academic papers for methodology, macro data for economic context.

It does NOT search the web. It only produces query strings.

// Receives both Agent 1 plan and Agent 2 data_summary
"Generate web search queries for the following
 research plan and quantitative findings.

 === RESEARCH PLAN (Agent 1) ===
 { qualitative_questions, presentation_angle, ... }

 === QUANTITATIVE FINDINGS SUMMARY (Agent 2) ===
 { top_performers, bottom_performers, key_stats }"

4
Web Researcher
FoundryAgent · Bing Search or Tavily MCP · Deployed in Azure AI Foundry portal
FoundryAgent Bing Search Tavily MCP Web access

Executes each search query from Agent 3 using real web search tools, reads the most relevant pages, and extracts sourced, factual claims with URLs. Every claim it returns must be traceable to a URL — this is what allows Agent 5 to cite sources without fabricating them.

It applies quality guardrails: dropping paywalled sources, unattributed opinion blogs, and content outside the relevant time window. It prefers major financial news outlets, research firms, government/central bank data, academic papers, and company filings. Limited to 2 sources per query to control response size.

▼ Input — Search Queries (from Agent 3)
// The full queries JSON from Agent 3
"Execute the following search queries and
 return a structured evidence pack.

 For each query:
 1. Search using your web search tool
 2. Identify 1-2 most relevant results
 3. Extract 2-4 specific, factual key_points
 4. Drop paywalled or unreliable sources"

5
Synthesizer
FoundryChatClient · No tools · Inline (no portal deployment needed)
FoundryChatClient No tools Pure reasoning

The final LLM agent. Receives all upstream outputs — the research plan, the quantitative findings, and the evidence pack — and assembles them into a slide_deck_spec JSON that the deterministic generate_pptx() function will render directly into PowerPoint slides.

Every factual claim in every slide must be backed by either a chart_id from Agent 2 or a URL from Agent 4 — no unsourced statements. This hallucination guardrail is enforced in the system prompt. It designs 6–10 slides in a logical narrative order: title → executive summary → chart slides → comparison tables → narrative context → sources.

▼ Input — All three upstream outputs
// Receives plan + findings + evidence in one message
"Build the slide_deck_spec JSON from:

 === RESEARCH PLAN (Agent 1) ===
 { intent, time_window, presentation_angle, ... }

 === QUANTITATIVE FINDINGS + CHART SPECS (Agent 2) ===
 { findings, chart_specs, data_summary, ... }

 === EVIDENCE PACK (Agent 4) ===
 { evidence_pack: [ {sources, key_points, ...} ] }"

f
render_chart_specs() + generate_pptx()
Local Python functions · matplotlib + python-pptx · No LLM involved
matplotlib python-pptx Deterministic No LLM

Two deterministic Python functions — not agents — that convert JSON specs into files. They always re-run (no checkpointing needed) since they are fast and side-effect-free.

render_chart_specs() reads the chart_specs list from Agent 2, renders each spec as a styled PNG using matplotlib, and saves to output/charts/<chart_id>.png. Supports line, bar, heatmap, and scatter chart types.

generate_pptx() reads the slide_deck_spec from Agent 5 and the chart PNG paths, then builds a fully styled PowerPoint file using python-pptx. It handles all six slide types with consistent navy/gold branding, headers, footers, bullet cards, tables, and chart images embedded at the correct positions.

▼ render_chart_specs() Input
// chart_specs list from Agent 2 findings JSON
[
  {
    "chart_id": "sector_cumulative_returns",
    "chart_type": "line",
    "series": [{"name": "IT", "data": [{"x":"2017-01", "y":0.05}, ...]}]
  }
]

// Returns: {"sector_cumulative_returns": Path("output/charts/sector_cumulative_returns.png")}
▼ generate_pptx() Input
// slide_deck_spec from Agent 5 + chart_paths dict
slide_deck_spec  →  Agent 5 JSON (slides array)
chart_paths      →  {"sector_cumulative_returns": Path("output/charts/...")}
output_path      →  Path("output/sp500_analysis_20260427_143022.pptx")

// Returns: the saved .pptx Path
  • Charts re-render every run — no checkpoint needed, always fast
  • If a chart_id in the deck spec has no matching PNG, the slide renders commentary-only (no broken image)
  • Slide branding: dark navy background, gold accent bar, white text, footer on every slide
  • Supports line, bar, heatmap, scatter for charts; title, exec_summary, chart_with_commentary, comparison_table, narrative, sources for slides
Checkpoint & Resume System

Single JSON file — one key per completed step

agent1_plan → agent2_findings → agent3_queries → agent4_evidence → agent5_deck_spec → pptx_path + chart_paths 
{
  "query":            "How did IT compare to Energy from 2017 to 2021?",
  "started_at":       "2026-04-27T14:30:22",
  "last_updated":     "2026-04-27T14:52:11",
  "completed_steps":  ["agent1", "agent2", "agent3"],  // agent4 not yet done
  "agent1_plan":      "{ ...research plan JSON... }",
  "agent2_findings":  "{ ...findings + chart_specs JSON... }",
  "agent3_queries":   "{ ...queries JSON... }",
  "pptx_path":        null,
  "chart_paths":      {}
}

On resume, the workflow reads this file, sees completed_steps: ["agent1","agent2","agent3"], skips those three agents, and picks up from Agent 4. The checkpoint file path is output/sp500_analysis_<run_id>.json — the run ID is the timestamp from when the run started.

CLI Reference
# Fresh run with default query python portfolio_risk_workflow.py # Custom query python portfolio_risk_workflow.py --query "Which sectors had the best Sharpe ratios 2017-2021?" # Resume interrupted run automatically (re-run same query) python portfolio_risk_workflow.py --query "Which sectors had the best Sharpe ratios 2017-2021?" # Resume by specific run ID (printed at start of every run) python portfolio_risk_workflow.py --run-id 20260427_143022 # Delete checkpoint for a query and restart from scratch python portfolio_risk_workflow.py --query "Which sectors..." --reset # Delete a specific run by ID and restart python portfolio_risk_workflow.py --run-id 20260427_143022 --reset
Output Files

output/sp500_analysis_<run_id>.json

The checkpoint file. Contains all agent outputs, completed steps, chart paths, and PPTX path. Grows with each completed agent. Used for resume and audit trail.

output/sp500_analysis_<run_id>.pptx

The final PowerPoint presentation. 6–10 slides with embedded charts, comparison tables, executive summary bullets, and a sources slide. Built by python-pptx — no LLM involved.

output/charts/<chart_id>.png

One PNG per chart_spec produced by Agent 2. Rendered by matplotlib at 150 DPI. Embedded in the corresponding chart_with_commentary slides in the PPTX.
